# Internationalization (i18n)

AutoMate is wired for translation but ships English-only. The infrastructure is in place so adding a new language is a translation task, not a refactor.

## Stack

- **`i18n-js`** — string lookup, ICU-style `{{var}}` interpolation, pluralization
- **`expo-localization`** — reads device locale at boot
- **`src/i18n/index.ts`** — exports `t()` (type-safe) and the underlying `i18n` instance
- **`src/i18n/locales/en.json`** — single source of truth for English strings; the type of `t`'s key is derived from this file's shape

## Using `t()` in screens and components

```tsx
import { t } from '@/src/i18n';

<Text>{t('settings.title')}</Text>
<Pressable accessibilityLabel={t('common.done')}>…</Pressable>
```

Keys are dot-notation paths into `en.json`. TypeScript autocompletes valid paths and errors on typos.

### Interpolation

```json
{ "settings": { "themeSetTo": "Set theme to {{label}}" } }
```

```tsx
t('settings.themeSetTo', { label: 'Dark' })
// → "Set theme to Dark"
```

### Pluralization

i18n-js doesn't do automatic plural selection from a single key in our setup. Use two keys and pick at the call site:

```json
{
  "settings": {
    "vehicleCountOne": "{{count}} vehicle",
    "vehicleCountOther": "{{count}} vehicles"
  }
}
```

```tsx
t(count === 1 ? 'settings.vehicleCountOne' : 'settings.vehicleCountOther', { count })
```

This is verbose but explicit — and survives translation to languages with more than two plural forms (Russian, Polish, Arabic) by adding more keys per locale.

### Composing translated fragments

Some strings interpolate already-translated fragments (e.g. the restore confirmation message embeds pluralized counts). Build the fragments with `t()` first, then pass them as params:

```tsx
const vehiclesPart = t(info.vehicleCount === 1 ? 'settings.vehicleCountOne' : 'settings.vehicleCountOther', { count: info.vehicleCount });
const message = t('settings.restoreConfirmMessage', { vehicles: vehiclesPart, /* ... */ });
```

## Naming conventions

Keys are namespaced by feature/screen:

- `common.*` — shared UI verbs/nouns (`done`, `cancel`, `save`)
- `tabs.*` — tab bar titles
- `tabsA11y.*` — tab bar accessibility labels (`"Dashboard tab"`)
- `settings.*` — the Settings screen
- `notFound.*` — the +not-found screen
- ...add a top-level namespace per screen as you migrate it

For accessibility labels that re-use the visible label, pass the same key. For accessibility labels that differ (e.g. richer description for screen readers), suffix the key with `A11y`.

## Adding a new language

1. Copy `src/i18n/locales/en.json` → `src/i18n/locales/<code>.json` (e.g. `es.json`)
2. Translate values (keys stay identical)
3. Register it in `src/i18n/index.ts`:

   ```ts
   import en from './locales/en.json';
   import es from './locales/es.json';

   const i18n = new I18n({ en, es });
   ```

4. Restart the app — `expo-localization` picks the active locale based on the device.

### AI-assisted translation workflow

Translate `en.json` in batches with Claude / GPT-4. Provide:

- The full source JSON (preserves key context)
- A short glossary for domain terms: "odometer," "MPG," "fill-up," "service log," "partial fill"
- The target locale and any tone notes
- Instruction to keep `{{interpolation}}` placeholders intact

Spot-check the highest-traffic strings (onboarding, settings, errors) and consider $30–50 of native-speaker review on Fiverr for the Play Store / App Store listing. UI strings can ship AI-only.

### Migration recipe per file

1. `import { t } from '@/src/i18n';`
2. For each user-visible literal, add a key to `en.json` under the right namespace.
3. Replace the literal with `{t('namespace.keyName')}`.
4. For interpolated literals (template strings), add `{{var}}` placeholders to the value and pass `t(key, { var })`.
5. For accessibility labels, prefer the same key as the visible label unless the screen-reader text is meaningfully different.
6. Run `npx tsc --noEmit` — TypeScript will catch any keys that don't exist.

## Caveats

- **No string change reactivity.** Locale is read once at boot. Adding a runtime language picker requires either an app reload or wrapping the app with a context provider that re-renders on change. Out of scope for MVP.
- **No RTL support yet.** Don't translate to Arabic / Hebrew until layout has been audited for `flexDirection: 'row-reverse'` and start/end positioning.
- **Dev-only error messages stay in English.** E.g. the developer "Test Error Boundary" crash message — these never reach end users.
